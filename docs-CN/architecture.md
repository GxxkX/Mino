# 架构

本文档介绍 Mino 的设计方式以及各组件之间的协作机制。
了解系统架构有助于故障排查、自定义配置和扩展系统功能。

## 系统概览

Mino 采用分层架构，在客户端、API 网关、服务层和数据存储层之间
实现了清晰的职责分离。各层通过定义良好的接口进行通信，使得替换
或修改某个组件时不会影响系统的其他部分。

系统由四个主要层次组成。

- **客户端层**包括所有面向用户的应用程序：智能手表、手机、Web
  和桌面客户端。
- **网关层**负责身份验证、速率限制、CORS、结构化日志记录和
  请求路由。
- **服务层**包含音频处理、语音转写、AI 分析、聊天、搜索和
  扩展等业务逻辑。
- **数据层**使用 PostgreSQL、Milvus、MinIO、Typesense 和
  Redis 存储信息。

## 客户端应用

Mino 支持四个客户端平台，每个平台都针对其设备进行了优化。

使用 Flutter 构建的**手表应用**专为可穿戴设备上的快速语音采集
而设计。它通过 WebSocket 将音频流式传输到后端，并在用户说话时
实时显示转写文本。离线时，应用会将录音本地存储在 SQLite 中，
待网络恢复后自动同步。该应用使用 Provider 模式进行状态管理，
并支持可配置的音频质量设置。

同样使用 Flutter 构建的**手机应用**提供了完整的界面，支持录音、
查看历史记录、管理记忆和任务，以及与 AI 助手聊天。它同时支持
音频文件的录制和播放。

**Web 应用**使用 Next.js 14 和 React 18 构建，提供响应式的
浏览器端界面。它包含移动端的所有功能，还提供扩展管理、全局搜索
（支持 Cmd+K 快捷键）和 MCP 配置等额外功能。该应用使用
Zustand 进行状态管理，并通过客户端 API 层与后端通信，支持
JWT 令牌自动刷新。

使用 Go 和 Wails 构建的**桌面应用**在 Windows、macOS 和
Linux 上提供原生性能。它以独立应用程序的形式提供与 Web 应用
相同的功能集。

## 后端服务

后端使用 Go 1.24 编写，采用 Gin Web 框架。它提供用于标准操作
的 REST API 和用于实时流式传输的 WebSocket 端点。应用程序编译
为单个二进制文件，连接到外部基础设施服务。

### API 网关

所有客户端请求首先经过 API 网关。网关承担以下职责。

- **身份验证**：使用 RSA RS256 签名验证 JWT 令牌。访问令牌
  15 分钟后过期，刷新令牌 7 天后过期。
- **速率限制**：使用基于 Redis 的滑动窗口计数器，以用户 ID
  和端点路径为键。默认限制为每分钟 100 个请求。当 Redis
  不可用时会优雅降级。
- **CORS**：为开发灵活性允许所有来源。
- **日志记录**：通过 logrus 进行结构化 JSON 日志记录，捕获
  状态码、方法、路径、查询参数、客户端 IP 和延迟信息。

### 音频服务

音频服务管理从录音到洞察的处理流水线。当 WebSocket 录音会话
结束时，音频服务接收完整的转写文本并编排处理工作流。

该服务在 PostgreSQL 中创建对话记录，然后调用 ExtractAgent
提取结构化信息（标题、摘要、待办事项和记忆要点）。提取的记忆
和任务通过数据库事务批量插入到各自的表中。提取完成后，服务
异步生成对话转写文本和提取的记忆的向量嵌入，然后将其存储在
Milvus 中以支持语义搜索。

该服务接受 PCM16 格式、16kHz 采样率的音频。支持 Opus 压缩
以减少移动端连接的带宽消耗。

### 语音转写服务

语音转写服务将语音转换为文本。它集成了语音转文本提供商，如
智谱 ASR 或自托管的 Whisper。该服务接收流式音频并实时返回
转写结果，使客户端能够在用户说话时即时显示文本。

您可以通过环境变量配置使用哪个提供商。该服务设计为支持多个
提供商，便于切换或添加新的转写引擎。

### AI 服务（双代理架构）

AI 服务使用 LangchainGo 实现了双代理架构。两个专用代理分别
处理不同的任务。

**ChatAgent** 驱动对话式 AI 助手。它接收用户消息以及从对话
历史中检索的上下文，并生成自然语言回复。系统提示词指示代理
基于历史对话进行回答、避免编造内容、自然地引用来源，并支持
多语言。

**ExtractAgent** 处理已完成的转写文本并提取结构化数据。它输出
严格的 JSON 格式，包含标题、摘要、待办事项和记忆要点。该代理
在每次录音会话完成后自动运行。

**SummarizeTitle** 是一个工具方法，根据第一条用户消息和助手
回复为聊天会话生成简短标题。

`LLMProvider` 接口定义了这些代理的契约。

```go
type LLMProvider interface {
    ChatAgent(ctx context.Context, userMessage string,
        retrievedContext string) (string, error)
    ExtractAgent(ctx context.Context, transcript string) (
        *StructuredResult, error)
    SummarizeTitle(ctx context.Context, userMessage string,
        assistantReply string) (string, error)
}
```

`EmbeddingProvider` 接口处理语义搜索流水线的向量嵌入生成。

```go
type EmbeddingProvider interface {
    EmbedQuery(ctx context.Context, text string) ([]float32, error)
    EmbedDocuments(ctx context.Context, texts []string) (
        [][]float32, error)
}
```

`LangchainLLMService` 实现了上述两个接口。它使用专用的嵌入
模型（通过 `LLM_EMBEDDING_MODEL` 配置），与聊天模型分开。
例如，使用智谱作为提供商时，聊天模型可能是 `glm-4.7-flash`，
而嵌入模型是 `embedding-3`。

LangchainGo 实现支持三种 LLM 提供商：OpenAI、智谱和 Ollama。
所有提供商均使用 OpenAI 兼容的 API 格式，通过
`LLM_PROVIDER`、`LLM_API_KEY`、`LLM_BASE_URL`、`LLM_MODEL`
和 `LLM_EMBEDDING_MODEL` 环境变量进行配置。

### 聊天服务

聊天服务管理与 AI 助手的对话交互。它提供基于会话的聊天功能，
支持对会话和消息的完整 CRUD 操作。

当用户发送消息时，服务执行以下步骤。

1. 将用户消息保存到数据库。
2. 使用 Milvus 语义向量搜索检索相关上下文。如果向量搜索不可用
   或未返回结果，则回退到 PostgreSQL 中的关键词匹配。
3. 将用户消息和检索到的上下文传递给 ChatAgent。
4. 将助手回复（包含来源引用）保存到数据库。
5. 如果是会话中的首次交互，使用 SummarizeTitle 自动生成
   会话标题。

聊天消息包含一个 `sources` 字段（在 PostgreSQL 中以 JSONB
格式存储），其中包含链接回原始对话的引用信息。

### 搜索服务

搜索服务使用 Typesense 提供跨对话和记忆的全文搜索功能。它管理
两个 Typesense 集合。

- **conversations**：索引字段包括标题、摘要和转写文本。
- **memories**：索引字段包括内容、类别和重要性。

该服务支持多集合搜索，并提供重建索引端点，将所有用户数据从
PostgreSQL 同步到 Typesense。当对话或记忆被创建时，文档会
自动同步。

### 向量存储服务

向量存储服务连接嵌入提供商和 Milvus，提供语义搜索能力。它承担
三项职责。

**索引**使用 `EmbeddingProvider` 为对话和记忆生成向量嵌入，
然后存储到 Milvus 中。对话文本在嵌入前会被截断至 6,000 个
字符，以符合 API 限制。记忆嵌入以批量方式生成以提高效率。

**搜索**将用户的查询文本进行嵌入，然后在 Milvus 中执行按
`user_id` 过滤的近似最近邻（ANN）搜索。`SearchAll` 方法同时
查询对话和记忆集合，按余弦相似度分数合并结果，并返回 Top-K
匹配项。

**删除**在对话或记忆被删除时从 Milvus 中移除对应的向量，保持
向量索引与 PostgreSQL 的一致性。

向量存储服务是可选的。如果 Milvus 或嵌入提供商在启动时不可用，
该服务将被设置为 `nil`，聊天服务会自动回退到关键词搜索。

### 扩展服务

扩展服务管理用户自定义的扩展。每个扩展包含名称、描述、图标、
启用状态和一个 JSON 配置对象。扩展按用户隔离，支持完整的
CRUD 操作。

## 数据存储

Mino 使用多个专用数据库来高效处理不同类型的数据。

### PostgreSQL

PostgreSQL 存储所有结构化数据。数据库模式包含九张表。

| 表名 | 用途 |
|------|------|
| `mino_users` | 用户账户和身份验证 |
| `mino_conversations` | 录音元数据和转写文本 |
| `mino_memories` | 提取的洞察和事实 |
| `mino_tasks` | 待办事项和任务条目 |
| `mino_tags` | 内容组织标签 |
| `mino_conversation_tags` | 多对多标签关联 |
| `mino_chat_sessions` | 聊天对话会话 |
| `mino_chat_messages` | 包含来源的单条聊天消息 |
| `mino_extensions` | 用户自定义扩展 |

所有表使用 UUID 主键。用户数据通过每个查询上的 `user_id`
外键实现严格隔离。

### Milvus

Milvus 是一个向量数据库，用于存储语义相似性搜索的嵌入向量。
后端管理两个集合。

- **conversations**：存储对话转写文本和摘要的向量嵌入。
- **memories**：存储提取的记忆内容的向量嵌入。

每个集合使用以下模式。

| 字段 | 类型 | 描述 |
|------|------|------|
| `id` | Int64（自增） | 主键 |
| `user_id` | VarChar(64) | 用于过滤的所属用户 ID |
| `source_id` | VarChar(64) | 对话或记忆的 UUID |
| `text` | VarChar(8192) | 与向量一起存储的原始文本 |
| `vector` | FloatVector | 嵌入向量 |

默认向量维度为 1024（与智谱 `embedding-3` 匹配）。如果使用
不同的嵌入模型，请相应调整维度。集合使用 IVF_FLAT 索引和
COSINE 相似度度量。

启动时，后端连接到 Milvus，创建缺失的集合，构建索引，并将其
加载到内存中以供搜索。如果 Milvus 不可达，后端会记录警告并在
没有向量搜索的情况下继续运行。

### MinIO

MinIO 提供 S3 兼容的对象存储，用于存储音频文件。音频文件以
唯一标识符存储，并与 PostgreSQL 中的对话记录关联。

> **注意：** MinIO 上传已配置，但音频上传流水线目前返回占位
> URL。完整的音频存储集成正在规划中。

### Typesense

Typesense 提供全文搜索功能。维护两个集合：`conversations`
（包含标题、摘要和转写文本字段）和 `memories`（包含内容、类别
和重要性字段）。搜索服务确保这些集合与 PostgreSQL 数据保持
同步。

### Redis

Redis 负责缓存、会话存储和速率限制。速率限制器使用以用户 ID
和端点路径为键的滑动窗口算法。速率限制计数器会自动过期。如果
Redis 不可用，速率限制器会优雅降级并允许请求通过。

## LangSmith 可观测性

Mino 集成了 LangSmith，用于 LLM 调用追踪和可观测性。通过
`LANGSMITH_TRACING` 环境变量启用后，所有 LLM 调用都会被追踪，
捕获输入/输出并异步发送到 LangSmith API。

该集成使用自定义回调处理器，实现了 LangchainGo 的回调接口。
追踪记录按 `LANGSMITH_PROJECT` 中配置的项目名称进行组织。

## 数据流

了解数据在系统中的流转方式有助于调试和优化。

### 录音流程

当您从任何客户端录制音频时，会执行以下步骤。

1. 客户端通过 `token` 查询参数传递 JWT 令牌，建立到
   `/v1/ws/audio` 的 WebSocket 连接。
2. 网关验证令牌并升级连接。
3. 客户端发送控制消息
   `{"type":"control","action":"start"}` 开始录音。
4. 客户端以二进制 WebSocket 消息的形式流式发送音频块（通常
   每 100 毫秒一次），音频数据经过 base64 编码。
5. 后端将音频块转发给语音转写服务。
6. 语音转写服务实时返回部分结果。后端将其推送给客户端，格式为
   `{"type":"transcript","text":"...","is_final":false}`。
7. 当客户端发送 `{"type":"control","action":"stop"}` 时，
   语音转写服务返回完整的转写文本。
8. 后端将转写文本发送给 AudioService 进行异步处理。
9. ExtractAgent 生成标题、摘要、待办事项和记忆。
10. 结构化数据存储到 PostgreSQL 中。记忆和任务通过事务批量
    插入。
11. 后端异步生成对话转写文本和提取的记忆的向量嵌入，然后将其
    存储到 Milvus 中以支持语义搜索。
12. 后端向客户端发送
    `{"type":"completed","conversation_id":"..."}` 消息通知
    处理完成。

### 聊天流程

当您与 AI 助手聊天时，会执行以下步骤。

1. 客户端向 `POST /v1/chat/sessions/:id/messages` 发送消息。
2. 网关对请求进行身份验证，并将其传递给聊天服务。
3. 聊天服务将用户消息保存到数据库。
4. 服务为用户查询生成向量嵌入，并在 Milvus 中对对话和记忆
   两个集合执行按用户 ID 过滤的 ANN 搜索。如果 Milvus 不可用
   或未返回结果，服务会回退到 PostgreSQL 中的关键词匹配。
5. ChatAgent 接收用户消息和检索到的上下文，然后生成包含来源
   引用的回复。
6. 助手回复及其来源信息被保存到数据库。
7. 如果这是会话中的首次交互，SummarizeTitle 会自动生成会话
   标题。
8. 回复连同引用信息一起返回给客户端。

### 搜索流程

当您搜索内容时，会执行以下步骤。

1. 客户端向 `GET /v1/search?q=keyword&limit=20` 发送查询。
2. 搜索服务在 Typesense 中执行多集合搜索，同时查询对话和
   记忆集合。
3. 结果按相关性排序返回，每个结果标明其来源集合。

## 安全性

Mino 实施了多项安全措施来保护您的数据。

所有通信使用 HTTPS 和 WSS 进行加密传输。密码使用 bcrypt
进行哈希处理，代价因子为 12 或更高。JWT 令牌使用 RSA RS256
密钥对签名，并设置了较短的过期时间。

用户数据严格隔离。每个数据库查询都按 `user_id` 过滤，确保
用户只能访问自己的录音、记忆、任务、聊天会话和扩展。

API 端点受到基于 Redis 的速率限制保护。系统按用户和端点跟踪
使用情况，默认在每分钟 100 个请求时阻止过多的请求。

## 扩展点

Mino 在多个方面支持扩展。

**Model Context Protocol (MCP)** 支持允许您将外部工具和服务
连接到 AI 助手。您可以通过 Web 界面的设置页面配置 MCP。

**应用扩展**提供可按用户启用或禁用的附加功能。每个扩展携带一个
JSON 配置对象，支持灵活的自定义。

**模块化服务架构**使得组件替换成为可能。例如，您可以替换语音
转写提供商（智谱 ASR、Whisper 或其他）、更换 LLM 模型或
提供商（OpenAI、智谱、Ollama），或添加新的存储后端，而不影响
系统的其他部分。

## 技术选型

Mino 中的每项技术都有其特定的选择理由。

选择 **Go** 作为后端语言，是因为其出色的性能、强大的并发支持
以及构建网络服务的成熟生态系统。Gin 框架提供了快速的 HTTP
路由和中间件支持。

**Flutter** 实现了手表和手机应用之间的代码共享，同时在两个
平台上提供原生性能。Provider 模式简化了手表应用的状态管理。

**Next.js 14** 为 Web 界面提供了 App Router，具备服务端渲染
能力和简洁的基于文件的路由结构。Zustand 提供轻量级的状态管理。

**PostgreSQL** 提供可靠的关系型数据存储和强大的查询能力。其
JSONB 支持可处理聊天消息来源和扩展配置等半结构化数据。

**LangchainGo** 提供了 Go 原生的 LLM 编排接口，通过统一的
API 支持多个提供商。结合 LangSmith，它为 AI 操作提供了完整的
可观测性。

**Typesense** 提供快速的全文搜索，具备拼写容错和相关性排序
功能，适合搜索对话转写文本和记忆内容。

**Milvus** 为语义检索流水线提供高性能的向量相似性搜索。其
IVF_FLAT 索引和 COSINE 度量支持快速的近似最近邻查询，其过滤
功能允许在向量层面实现按用户的数据隔离。
